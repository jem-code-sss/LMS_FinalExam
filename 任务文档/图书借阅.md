**校园图书借阅管理系统的设计与实现**

# 一、课程设计目的

本实践题目以校园图书借阅管理为背景，要求学生围绕图书、读者和借阅记录实现完整业务流程，训练需求规格定义、数据一致性维护、日期计算、文件持久化和测试验证能力。

通过本实践，学生应达到以下目标：

1. 能够根据图书借阅场景提炼系统需求，设计合理的数据结构和程序模块。

2. 能够设计Book、Reader、BorrowRecord、LibrarySystem等核心数据类型。

3. 能够实现图书添加、读者管理、借书、还书、查询、保存与读取等业务流程。

4. 能够实现日期合法性判断、借阅天数计算、逾期判断和排序统计。

5. 能够通过测试用例验证数据一致性，并记录AI辅助开发过程。

# 二、设计任务及要求

## 1. 项目背景与程序形态

项目采用C/C++控制台程序形式，不要求图形界面。学生应完成需求分析、概要设计、详细设计、编码实现、调试测试、版本迭代、报告撰写和答辩说明。

## 2. 基本数据要求

|   |   |   |
|---|---|---|
|**数据项**|**说明**|**最低约束**|
|书号bookId|图书唯一编号|不得重复，不能为空|
|图书信息|书名、作者、类别、总册数、可借册数|可借册数不得大于总册数|
|读者编号readerId|读者唯一编号|不得重复，不能为空|
|借阅记录|书号、读者编号、借阅日期、归还日期、状态|归还日期不得早于借阅日期|
|借阅次数|图书被借阅次数|用于排序或热门图书统计|

## 3. 建议设计的核心数据类型

|   |   |   |
|---|---|---|
|**类/结构体**|**主要成员或职责**|**最低要求**|
|Date|日期合法性判断、借阅天数计算。|能处理闰年和日期先后关系。|
|Book|保存图书信息和库存状态，提供canBorrow、borrowOne、returnOne等方法。|库存更新应保持一致。|
|Reader|保存读者基本信息。|读者编号唯一。|
|BorrowRecord|保存每次借阅和归还状态。|不能只修改图书数量而丢失记录。|
|LibrarySystem|统一管理图书、读者和借阅记录。|负责菜单、查询、保存读取和业务流程。|
|Reader及派生类（进阶）|可将Reader作为读者基类，派生StudentReader、TeacherReader等，通过虚函数返回最大可借册数、最长借阅天数或逾期规则。|至少定义1个虚函数，并在借书、还书或逾期判断流程中调用。|

# 六、文件保存与读取格式要求

为保证学生实现口径一致、教师能够使用统一样例文件复测，基础功能的保存与读取应按以下格式实现。学生可以在进阶功能中扩展其他格式，但不得影响本节规定的基础格式。

|   |   |
|---|---|
|**项目**|**统一要求**|
|文件数量|基础版建议使用3个CSV文件分别保存图书、读者和借阅记录：books.csv、readers.csv、borrow_records.csv。|
|编码|建议使用 UTF-8 文本文件。|
|分隔符|英文逗号 , 分隔字段；每条记录占一行。基础版字段内容不得包含英文逗号，避免解析歧义。|
|图书字段顺序|bookId,title,author,category,totalCount,availableCount,borrowTimes|
|读者字段顺序|readerId,name,phone|
|借阅记录字段顺序|recordId,bookId,readerId,borrowDate,returnDate,status|
|日期格式|borrowDate、returnDate 统一使用 yyyy-mm-dd；未归还记录的 returnDate 可填写 NONE。|
|状态字段|status 建议使用 BORROWED 表示未归还，RETURNED 表示已归还。|
|表头|允许首行为表头；读取时如发现表头，应跳过表头。|

|---|---|---|
|**读取情形**|**处理规则**|**验收证据**|

# 七、开发规范要求

1. 变量、函数、类名应清晰，函数名应体现功能。

2. 不得将全部功能写在main()函数中，至少进行函数划分，鼓励拆分为.h和.cpp文件。

3. 菜单应清晰，输入错误应有提示，操作成功或失败应有反馈。

4. 文件不存在、文件为空、文件格式错误时应能给出提示并避免程序崩溃。

5. 每个版本均需测试，不允许只在最终版本测试。

6. 使用动态内存时应正确释放资源；使用STL时应说明使用原因。
